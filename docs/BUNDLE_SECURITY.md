# SCHF Bundle Security

**Version:** 1.0.0

---

## 1. Integrity

### Checksums

Every bundle contains `checksum.sha256` with SHA-256 hashes for every file in the bundle. The checksum file itself is verified by its entry in `manifest.json`.

**Validation flow:**
1. Extract bundle to temporary directory
2. Read `checksum.sha256`
3. For each entry, compute SHA-256 and compare
4. Any mismatch → bundle rejected

### Path Traversal Protection

During extraction, all file paths are sanitized:
- Reject paths with `..`
- Reject absolute paths
- Reject paths with drive letters (`C:`)
- All paths must be relative to bundle root

## 2. Signatures

### Purpose

Digital signatures ensure the bundle was generated by an authorized source and has not been tampered with since signing.

### Supported Algorithms

| Algorithm | Extension | Minimum key size |
|-----------|-----------|------------------|
| RSA-SHA256 | OpenSSL | 2048 bits |
| Ed25519 | libsodium | 256 bits |

### Signing Flow

1. Generate `checksum.sha256` from all bundle files
2. Sign the `checksum.sha256` contents with the private key
3. Store signature in `signature.sig`
4. Optionally embed `signing-key.pub` in the bundle

### Verification Flow

1. Validate bundle (checksums, manifest, JSON)
2. Extract `signature.sig`
3. Extract or provide `signing-key.pub`
4. Verify signature against `checksum.sha256` contents
5. Failed verification → bundle flagged, import blocked

### Key Management (Recommended)

- Signing keys should be per-client or per-release
- Private keys stored in secure vault/HSM
- Public keys distributed via secure channel
- Key rotation supported (bundle version + key ID in manifest)

## 3. Data Safety

### No Credentials

Bundles **must not** contain:
- Database credentials
- API keys or tokens
- Encryption keys
- Passwords or secrets

### No Executable Code

Bundles **must not** contain:
- PHP, Python, JavaScript, or any script files
- Binary executables
- Shell scripts
- SQL or database dump files

### No Path Traversal

All file paths in the ZIP are validated against:
- Directory traversal (`../`)
- Absolute paths (`/etc/`, `C:\`)
- Symbolic links (not extracted)

## 4. Validation Rules

### Required Checks (always performed)

- [x] File existence (all 13 required files)
- [x] SHA-256 checksum verification
- [x] Manifest field completeness
- [x] JSON parseability
- [x] Version format validation
- [x] UUID format validation

### Conditional Checks

- [ ] Signature verification (if `signature.sig` present)
- [ ] Schema validation (if schema files available)

### Import Guards

- Bundle must pass all required checks before preview
- Preview must be confirmed before import
- Import is wrapped in a database transaction
- Transaction rollback on any failure

---

## 5. Gitleaks

The SDK includes `.gitleaks.toml` rules to prevent accidental commits of:
- Private keys (RSA, Ed25519)
- Database connection strings
- API tokens and secrets
- .env files

Run before every commit:

```bash
gitleaks detect --source . --verbose
```

---

## 6. Security Checklist

Before releasing a bundle to production:

- [ ] All checksums verified
- [ ] Signature applied and verified
- [ ] No credentials in bundle files
- [ ] No personally identifiable information (PII) beyond what is required
- [ ] Bundle validated by `schf bundle doctor`
- [ ] Bundle reviewed with `schf bundle inspect`
- [ ] Gitleaks scan passed (0 leaks)
